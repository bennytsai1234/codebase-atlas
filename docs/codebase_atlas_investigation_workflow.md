# Codebase Atlas 調查工作流程

## 目的

- 當使用者想了解某個規則、模板、模式、文件敘事或品質約束是否合理時，使用此流程。
- 此流程用於釐清事實、假設與未知；不要在未確認前直接改檔。
- 不要為一般調查重新執行 Codebase Atlas。

## 工作流程

1. 保留使用者的調查問題。
1. 開啟 `codebase_atlas_index.md`，選出最可能相關的模組或邊界。
1. 閱讀相關 module docs 的範圍、依賴、下游影響、主要流程與已知風險。
1. 檢查最小必要檔案，例如 README、`SKILL.md`、references、templates 或 self-bootstrap docs。
1. 分開記錄事實、推論、未知與需要使用者決策的地方。
1. 若問題跨模組，說明主要模組與邊界模組的角色。
1. 若下一步需要改檔，建議切換到 bug、feature、optimization 或 refactor workflow。
1. 若沒有修改檔案，總結調查結果並說明無需 commit；若使用者明確要求修改，依交付政策收尾：commit and push。

## 調查摘要

在結論前提供白話摘要：

- **問題**：這次要釐清什麼？
- **地圖**：哪些模組相關？
- **事實**：檔案與文檔顯示了什麼？
- **推論**：哪些是根據事實推得，但仍需要判斷？
- **未知**：哪些仍缺少證據？
- **下一步**：若要改檔，應使用哪個 workflow？

若下一步會修改檔案，加入白話修改前／修改後摘要，並在使用者明確確認後才開始後續編輯。

## 交付政策

commit and push
